# CS255-SystemAnalysisAndDesign
CS255 Course SNHU
System Design and Analysis Reflection
Project Summaries and Client Needs In the first half of this course, I worked on the DriverPass project. The client was Liam, who owns a driving school called DriverPass, along with his IT officer, Ian. They needed a management system that worked both online and offline to help their students pass their local DMV driving tests. I had to design a system that could handle scheduling on the road training in rigid two hour blocks, tracking student progress on practice tests, and managing a fleet of 10 cars. The system also needed to pull live updates from the DMV for rule changes and let Liam export activity reports to Microsoft Excel for his offline work. 
For the second project, I shifted to technical system modeling and created UML diagrams for an ATM Cash Withdrawal system. This part of the class required me to turn business logic into technical blueprints. I mapped out the sequence of events, user interactions, and database connections needed to securely process cash withdrawals.
Areas of Success I feel like I did a really good job breaking down complex requirements into organized lists. For the DriverPass Business Requirements Document, I clearly separated the functional requirements from the nonfunctional ones. I successfully recognized that while the system must functionally allow students to book lessons, it nonfunctionally needed strict role based access and cloud hosting to keep data secure without making extra work for the IT officer. 
In the second project, I excelled at building the actual UML diagrams. Making the activity, sequence, and class diagrams took a lot of attention to technical detail. I successfully mapped out the logic of the ATM system to make sure things like insufficient funds and wrong PIN entries were handled properly. Getting the specific UML rules right, like adding the activation boxes and the exact multiplicity labels, is something I am really proud of.
Areas for Revision If I could revise one part of my work, I would definitely focus on the system limitations section of the DriverPass document. I originally focused way too much on technical limitations, like the system going down if the cloud hosting provider had an outage. While that is true, I could improve this section by including more business level limitations. For instance, I would analyze how the software is supposed to handle scheduling conflicts if a car breaks down or a driver calls in sick. Expanding on these real world problems would make the document much more useful for Liam and Ian. Also, for the UML diagrams, I would spend more time visualizing the spacing of my class attributes on the first draft so I would not have to do as much resizing and cleanup at the very end. 
Interpreting and Implementing User Needs Interpreting user needs means looking past just the basic software features to figure out what the client actually has to do every day. For DriverPass, this meant understanding the different jobs at the company. Liam needed high level reports, the secretary needed an easy form to register students over the phone, and the IT officer needed a simple way to reset passwords without touching the code. I implemented these needs into the design by specifying a dashboard interface and using a third party cloud host for database security. 
It is incredibly important to consider the user when designing because a technically perfect program is completely useless if it does not solve the client's actual problems. If the software is too confusing for the secretary to use or fails to track the specific two hour lesson blocks that DriverPass relies on, the whole project fails. It does not matter how good the code is if the users cannot actually do their jobs. 
Approach to Software Design and Future Strategies
My approach to designing software starts with business analysis. I think you really have to map out the users, their goals, and the constraints before you even think about writing code or building databases.
In the future, the main strategy I will use is creating visual UML models much earlier in my process. Building the use case and activity diagrams helped me see logical gaps that are really easy to miss when you are just typing out text requirements. Moving forward in my computer science courses, I plan to use tools like Lucidchart to diagram business logic side by side with the client. Using visual strategies like this will help me confirm user requirements right away, which ensures the final system actually matches what the client expects before any development begins.
